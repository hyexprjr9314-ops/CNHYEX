# Codex 작업 규칙

## 시작 순서

1. `PROJECT_HANDOFF.md`에서 현재 상태와 기술 부채를 확인한다.
2. UI 작업이면 `DESIGN.md`, 데이터·API 작업이면 `docs/ARCHITECTURE.md`를 읽는다.
3. `rg`로 관련 화면·함수·SQL만 먼저 찾는다.
4. 전체 `index.html` 정독은 교차 화면 영향이 실제로 있을 때만 한다.

## 변경 원칙

- GitHub `main`을 운영 기준으로 사용하며 직접 수정하지 않는다.
- `agent/<작업명>` 브랜치와 Pull Request를 사용한다.
- 사용자 요청과 무관한 기존 동작·스타일·데이터 구조를 변경하지 않는다.
- 브라우저 저장소에는 글자 크기 같은 UI 환경설정만 허용한다.
- 사용자·평가·배정·점수·공개 상태는 Supabase를 단일 기준으로 유지한다.
- 권한 검사는 화면 숨김이 아니라 RLS와 서버 API에서도 수행한다.
- 새 의존성은 필요성이 명확하고 사용자가 동의한 경우에만 추가한다.

## 보안

- 비밀키, 비밀번호, 토큰, 실제 임직원 CSV 및 평가 내보내기를 커밋하지 않는다.
- `SUPABASE_SERVICE_ROLE_KEY`는 Vercel 서버 환경변수로만 사용한다.
- 사용자 입력을 HTML에 삽입할 때 `escapeHtml` 또는 DOM 안전 API를 사용한다.
- 관리자 API는 Supabase 세션과 서버 측 관리자 권한을 모두 검증한다.

## 검증

- 변경된 인라인 스크립트를 `new Function(...)` 등으로 구문 검사한다.
- 중복 DOM `id`가 없는지 검사한다.
- 변경한 UI 상태의 활성·비활성·오류·빈 상태를 확인한다.
- SQL 변경은 새 마이그레이션으로 추가하고 기존 적용 파일을 다시 쓰지 않는다.
- PR의 Vercel Preview가 성공하기 전에는 병합하지 않는다.
- 병합 후 운영 주소가 HTTP 200을 반환하고 변경 코드가 제공되는지 확인한다.

## 인수인계 응답

완료 보고에는 다음만 간결하게 남긴다.

- 변경된 기능
- 검증 결과
- PR 번호와 병합 여부
- 운영 배포 여부
- 남은 위험 또는 수동 확인 사항
